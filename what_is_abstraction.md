### Разминочное - 1.

_Привести примеры, когда реализация одинакова, но спецификация разная, и поэтому реализации должны поддерживаться и развиваться по-разному._

#### Пример №1.

Есть два микросервиса, А и B. В обоих применяются паттерны message и outbox. Реализация этих паттернов для двух севрисов изначально была одинаковая &ndash; сохраняем входящее сообщение, извлекаем из БД с использованием scheduled job, пытаемся обработать, отправляем в другую систему, используя ее API.
Но принципиальная разница в том, что сервис А просто преобразовывает сообщение и отправляет в другую систему, а сервис B перед отправкой выполняет сложную логику бухгалтерского учет по данным из входящего сообщения. 

Но при реализации было принято решение сделать общую библиотеку для двух сервисов. И реализация со временем стала обрастать различными условиями, параметрами, нужными для сервиса B. Но они были совсем не нужны в сервисе A, и даже ломали его логику время от времени.

Поэтому спустя 2 года пришлось убирать зависимость на общую библиотеку из сервиса A. Но т.к. времени как всегда мало, то не было возможности провести полноценный рефакторинг и выделить именно тот код, который нужен в сервисе  A. Просто скопировали классы из библиотеки в сервис A, занеся кучу лишнего функционала, который там не используется. Но хоть процессинг не сломали. 


#### Пример №2.

Есть два микросервиса A и B, A отправляет json-сообщение в B. Схема Json для сообщения и соответствующий DTO, в который десериализуется сообщение (пусть будет AbstractMessage), хранится в общей библиотеке.

Сервис B после обработки отправляет сообщение в сервис C, и оно по формату такое же, как AbstractMessage. И разработчики сервиса B использовали AbstractMessage для транспорта между B и C.
Но хотя формат сообщений совпадает, у них разное назначение: AbstractMessage &ndash; это контракт между разными системами A и B, которые поддерживают разные команды. А сервисы B и C &ndash; находятся в одном контуре и поддерживаются одной командой, и этот контракт команда может менять в любой момент по своему желанию для удобства.

И вскоре разработчики сервиса B решили добавить в AbstractMessage новый атрибут, который нужен в сервисе C. И это сломало API в сервисе A.

#### Пример №3.

Сервис принимает сообщения разного типа и обрабатывает их. Все типы объединяет то, что результат обработки должен быть отправлен в систему X.

Увидела этот код на ревью &ndash; предполагалось, что любое сообщение будет сначала обрабатываться в родительском методе Handler#handle. Ведь любое сообщение, вне зависимости от типа, требуется десериализовать, сохранить, обработать, и отправить в сервис X. И для каждого типа сообщений будет свой потомок класса Handler, который в переопределенном методе handleMessage будет как раз делать обработку, специфичную для конкретного типа сообщений.


```java

public abstract class Handler {

    public void handle(String strMsg) {
        Message message = desesialize(strMsg);
        repository.save(message);
        //...
        XMessage xMsg = handleMessage(message);
        apiX.sendMessage(xMsg);
    }

    @Override
    public XMessage handleMessage(Message msg) {
        //...
    }
}
```


Но для обработки сообщений разного типа требуется совершенно разная логика. И получается, что ради того, чтобы сохранить реализацию handle одинаковой для сообщений разного типа, нам придется в метод handleMessage запихивать абсолютно все. А там может быть просто преобразование Message в XMessage, а могут быть вызовы сторонних API. И кажется, что не очень правильно такую разную логику делать в одном методе handleMessage. 

### Разминочное - 2.
_При выполнении задания "Увеличиваем полиморфность кода" &ndash; не скатывались ли вы в боксинг?_

В том задании я разобрала пример, когда у нас есть 2 реализации для обработки входящих и исходящих сообщений. Я объединила эту логику в один метод.
Там выходило, что для единообразия реализации мы в таблицу для входящих сообщений мы добавляем счетчик попыток, который используется для исходящих сообщений, но устанавливаем его равным 1, поэтому логика для входящих не меняется.

Но, допустим, что выходящее сообщение мы хотим валидировать относительно схемы, а исходящее не надо, т.к. оно гарантировано создано правильно.
Или при отправке сообщения мы хотим не просто писать ошибку в лог и повторять n раз, а сделать более умную реализацию &ndash; если получили ошибку с кодом 4xx от сторонней системы, то не повторяем обработку сообщения, а если с кодом 5xx &ndash; то повторяем. А для входящего это не нужно.

Поэтому вероятно, реализация бы действительно скатилась в боксинг.

### Разминочное - 3.
_Как часто в среднем в час вам приходится заглядывать в другие классы, чтобы разобраться со смыслом функций, классов? Как уменьшить (свести к нулю) это число?_

Приходится постоянно заглядывать, думаю, это как минимум двузначное число раз в час.

На этой неделе, например, как раз была рабочая задача, когда пришлось разбираться с новым для меня кодом.
Нужно было расширить логику обработки сообщения, которое приходит в микросервис из другой системы. Сейчас сообщение может быть одного типа, а будет еще 3 других типа.
За час я переключалась между десятком классов много раз. Составила диаграмму вызовов, начиная от Kafka-слушателя, до сервиса, который отправляет обработанное сообщение в стороннюю систему.

У нас используется Spring, и наша архитектура как раз очень похожа на то, что названо в материале "мартинфаулершина".

##### Вариант 0. Изменить архитектуру проекта.

Возможно, радикальный вариант уменьшить число переключений &ndash; уйти от нашей анемичной доменной модели, где доменные объекты просто используются для маппинга на таблицы БД. И новые системы проектировать так, чтобы бизнес-логика содержалась в объектах доменной модели, а не только в слое сервисов.

Если не менять архитектуру, то возможно варианты могут быть такие:

##### Вариант 1. Писать больше комментариев

А то когда имеетcя множество классов слоя Service, которые вызывают другие классы, да еще они параметризованы, необходимо постоянно ходить туда-сюда по цепочке вызовов, чтобы хотя бы примерно понять, что происходит.
Например, у нас есть немало таких классов, где хорошо бы написать Javadoc, и хотя бы привести примеры, какими могут быть параметры класса:
```java

class SomeHandler<T extends Message, P, Q, R extends BasicRepository, S extends MessageProcessingService> extends BasicHandler<...>{
    SomeProcessor processor;
    Q handle (T msg) {
        //...
        processor.process(message);
    }
}
```
##### Вариант 2. Более понятные названия классов.

Еще, возможно, проблема в том, что непонятно по названию класса, что он хотя бы примерно делает.  Например, совершенно неясно, чем InboundMessageProcessor отличается от InboundMessageHandler и от InboundMessageService, пока не заглянешь в них.

По факту оказывается, что названия &ndash; это просто какие-то общие слова, которые примерно-обобщенно соответствуют логике:
- InboundMessageProcessor &ndash; десериализует сообщение и валидирует относительно json схемы.
- InboundMessageHandler &ndash; превращает DTO в Entity по правилам бизнес-логики.
- InboundMessageService &ndash; здесь основная бизнес-логика.
