### 1
В приложение приходят сообщения о регистрации нового продукта.
Тестируется метод `processEvents()` класса `AssetProcessingService`.
Ожидаемый абстрактный эффект для данного теста &ndash; для всех событий из таблицы inbox с типом ASSET_REGISTRATION и статусом NEW:
- создается запись в таблице asset
- создается 3 outbox-события для отправки в сторонние системы


Для проверки создания outbox событий здесь 3 раза вызывается метод `findByTypeAndStatusAndNextExecutionTimeBefore` класса `OutboxRepository`. И на мой взгляд, здесь происходит тестирование реализации этого метода, вместо тестирования абстрактного эффекта.

В реальной ПРОМ-среде используется БД Postgres, а в тесте &ndash; in-memory БД H2.
Метод `findByTypeAndStatusAndNextExecutionTimeBefore` - реализован через native query в Spring Data. Мы полагаемся на то, он будет одинаково работать и для H2 и для Postgres.
Также в запросе передается время (nextExecutionTime). Но на ПРОМ база данных может сохранять даты в другом часовом поясе, нежели это делает тестовая БД H2.
И тесты могут запускаться с другой таймзоной, нежели в ПРОМ.
И еще один момент, что в данном тесте мы не сможем отловить ситуацию, когда создалось больше outbox-сообщений, чем нужно.

```java
void processEvents__shouldProcessAssetEventAndCreateAsset() {
    // Создается inbox-сообщение с запросом о регистрации нового продукта и сохраняется в in-memory БД H2
    InboxtEntity inbox = inboxRepository.save(InboxEntity.builder()
            .inboxType(inboxType.ASSET_REGISTRATION)
            .status(inboxStatus.NEW)
            .source(Source.XXX)
            .data(d)
            .attempts(0)
            .nextExecutionTime(EXECUTION_TIME)
            .build());

    assetProcessingService.processEvents();

    //проверяем, что входящее сообщение перешло в статус DONE
    inbox = inboxRepository.findById(inbox.getId()).orElseThrow();
    assertThat(inbox).returns(inboxStatus.DONE, inboxEntity::getStatus);
    AssetEntity asset = assetRepository.findByCode(ASSET_CODE).orElseThrow();
    assertThat(asset)
            .extracting("status", "code")
            .containsExactly(assetStatus.CHECK_PASSED, ASSET_CODE);

    List<OutboxEntity> outboxes = outboxRepository.findByTypeAndStatusAndNextExecutionTimeBefore(ASSET_SETUP, 1, Instant.now());
    assertThat(outboxes).hasSize(1);

    outboxes = outboxRepository.findByTypeAndStatusAndNextExecutionTimeBefore(EVENT_RESPONSE, 1, Instant.now());
    assertThat(outboxes).hasSize(1);

        outboxes = outboxRepository.findByTypeAndStatusAndNextExecutionTimeBefore(SEND_ASSET_TO_BLOCKCHAIN, 1, Instant.now().plus(delay, ChronoUnit.SECONDS));
    assertThat(outboxes).hasSize(1);
}
```

### 2
В сервис приходит сообщение с ценой финансового актива (котировка).

Абстрактный эффект, который хотели бы проверить в данном тесте - при получении котировки создается в базе данных создается запись в таблице quote, содержащая данные из входящего сообщения.

Но в приложении сообщение с котировкой вычитывается из Kafka. А в тесте эмулируется через вызов метода `sendMessage`. Через mockMvc (вспомогательный класс из Spring Test, который позволяет тестировать HTTP-эндпоинты без запуска реального сервера), вызывается эндпойнт, который принимает сообщение. Но эндпойнт тоже используется только в тестах.
Возможны расхождения между тем, как будет обработано сообщение, переданное через http-запрос в тесте, и вычитанное из Kafka в реальной среде. 
Или сообщение может прийти в другой котировке, не в UTF-8, как заложено в sendMessage.
Также возможна проблема в objectMapper, в приложении он может быть настроен по-другому, нежели в тесте.

```java
void shouldCreateQuote(String pathToMsg, String endpoint) throws Exception {
        // отправка сообщения c котировкой
        Message msg = sendMessage(Message.class, path, endpoint);

        // getList - вызывает тестовый эндпойнт, который возвращает все объекты QuoteEntity
        // По сути, тут мы тестируем реализацию метода getList. 
        // К тому же объект QuoteEntity не обязательно будет корректно десериализоваться, ведь он предназначен только для ORM маппинга
        List<QuoteEntity> quotes = getList(QuoteEntity.class, "/quotes");
                List<QuoteEntity> expectedQuotes = map(message.getPayload());

        assertThat(quotes).usingRecursiveFieldByFieldElementComparator(recursiveCompConfigurationBuilder)
                .usingComparatorForType(BigDecimal::compareTo, BigDecimal.class)
                .containsExactlyInAnyOrderElementsOf(expectedQuotes);
    }

protected <T> T sendMessage(...) throws Exception {
    String content = IOUtils.resourceToString(resource,Charset.forName("UTF-8"));
    //...
    T message = objectMapper.readValue(content, messageType);
    mockMvc.perform(post(path).content(content).contentType(APPLICATION_JSON_VALUE))
            .andExpect(status().isOk())
            .andReturn();
    return message;
}

protected <T> List<T> getList(...) throws Exception {
    MvcResult mvcResult = mockMvc.perform(get(url, params))
            .andExpect(status().isOk())
            .andReturn();
    List<T> list = JsonUtils.parseList(mvcResult.getResponse().getContentAsString(UTF_8), objectType);
    return list;
}
```

### 3
В сервис приходит сообщение с данными о регистрации нового продукта.

Абстрактный эффект, который хотим протестировать: при получении сообщения отправляется запрос в блокчейн о регистрации
нового датафида.

В данном тесте поднимается testcontainer с образом блокчейн.
Затем мы получаем из нашей БД идентификатор датафида в Блокчейн, и по нему пытаемся найти датафид в блокчейн.
Но мы тестируем реализацию, т.к. с новой версией блокчейна может поломаться и образ, и тест.

```java
void shouldCreateDataFeedInBlockchain(String pathToMasg) throws Exception {
    FeedConfigEntity expectedConfig = createExpectedConfig();
    DataFeed expectedDataFeed = getDataFeed(expectedConfig);

    //эмулируется получение сообщения о регистрации через тестовую ручку (как в примере 1).
    sendMessage(AssetRegistrationMessage.class, pathToMasg, "/asset/registaration");

    //получаем из базы данных созданный объект FeedConfigEntity, чтобы у него получить bchFeedId
    FeedConfigEntity feedConfig = feedConfigRepository.findByLabel(expectedConfig.getLabel()).get(0);

    //вызов testcontainer где развернут blockchain 
    DataFeed actualFeed = getDataFeed(feedConfig.getBCHFeedId());

    //проверка, что в БЧ создалось то, что мы ожидали
    assertThat(actualFeed).isEqualTo(expectedDataFeed);
}

protected DataFeed getDataFeed(String feedId) {
    return getOrThrow(chain.getDataFeed(feedId), DataFeed.class);
}
```

Вместо того, чтобы поднимать тестконтейнер с образом блокчейн, можно использовать мок класса, который делает вызов в
блокчейн.

```java
@McokBean
BlockchainDao blockchainDao;

void shouldCreateDataFeedInBlockchain(String pathToMasg) throws Exception {
    FeedConfigEntity expectedConfig = createExpectedConfig();
    DataFeed expectedDataFeed = getDataFeed(expectedConfig);

    // when
    when(blockchainDao.listDataFeeds()).thenReturn(List.of());

    // do
    sendMessage(AssetRegistrationMessage.class, pathToMasg, "/asset/registaration");

    // verify
    verify(blockchainDao, times(1)).registerDataFeed(eq(expectedDataFeed.description()));
}
```
