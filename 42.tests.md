Имеется приложение - блокчейн-оракул. Оно поставляет котировки в датафиды в блокчейне. 

Фича: новый датафид для отображения на UI.

Описание: 
Созается новый датафид для отображения информации на UI. Он не зарегистрирован в смарт-контракте. 
Когда 3-d party система присылает значение котировки для него, оно не должно сразу отправляться в блокчейн, как для остальных датафидов. 
Вместо этого просто сохраняем значение в БД, и отправляем в 9:00 следующего рабочего дня. 
Для реализации добавляем в таблицу, где хранятся параметры датафидов (datafeeds), колонку is_special_publication - по умолчанию false, а для специального датафида - true.

Свойства, задающие корректность поведения для датафидов с параметром is_special_publication = true:

#### 1
При получении сообщения с котировкой:
  - сохраняем сообщение в таблицу inbox,  
  - обновляем запись в таблице actual_quote 
  - создаем запись в таблице historical_quote

```java

    private final static Payload payload = Payload.builder()
            .unstrumentId("ABCD.123")
            .ask("3567.89")
            .bid("3789.09")
            .updateTime("20-06-2026T12:34:10.000")
            .updateUserId("46752976").build();

    private final static Message<Payload> msg = new Message<>(payload);

    @Test
    void test_inbox_creation() {
        inboxMessageHandler.handle(msg);

        verify(inboxRepository, times(1)).save(inboxCaptor.capture());
        InboxEntity inbox = inboxCaptor.get();
        assertThat(inbox).isEqualTo(expectedInbox);
    }

    @Test
    void test_actual_quote_creation() {
      inboxMessageHandler.handle(msg);

      verify(actualQuoteRepository, times(1)).save(actualQuoteCaptor.capture());
      ActualQuoteEntity actual = actualQuoteCaptor.get();
      assertThat(actual).isEqualTo(expectedActual);
    }


    @Test
    void test_historical_quote_creation() {
      inboxMessageHandler.handle(msg);

      verify(historicalQuoteRepository, times(1)).save(historicalQuoteCaptor.capture());
      HistoricalQuoteEntity historical = historicalQuoteCaptor.get();
      assertThat(historical).isEqualTo(expectedHistorical);
    }

```

#### 2
При получении сообщения с котировкой не должно быть вызова в блокчейн и не должен быть создан outbox.

```java
    @Test
    void test_no_blockchain_call_no_outbox() {
        inboxMessageHandler.handle(msg);
        verify(blockchainClient, times(0)).pushFeedValue(any());
        verify(outboxRepository, times(0)).save(any());
    }
```

#### 3
При вызове метода отправки (который запускается по расписанию) происходит отправка сообщения в блокчейн и создание outbox.

```java
    @Test
    void test_blockchain_call() {
        specialPubModeScheduler.execute();

        verify(blockchainClient, times(1)).pushFeedValue(datafeedValueCaptor.capture());
        DatafeedValue value = datafeedValueCaptor.get();
        assetThat(value).isEqualTo(expectedValue);
    }

    @Test
    void test_outbox_creation() {
        specialPubModeScheduler.execute();
        
        verify(outboxRepository, times(1)).save(outboxCaptor.capture());
        OutboxEntity outbox = outboxCaptor.get();
        assetThat(outbox).isEqualTo(expectedOutbox);
    }
```

### 4
При создании новых датафидов значение по умолчанию is_special_publication = false.

Способ: интеграционные тесты с БД. 

### 5
Для данного датафида отправка значения в блокчейн и создание outbound_message происходит раз в сутки именно в 9:00.

Способ: через анализ лог-файлов, outbound сообщений в БД на тестовом стенде, проверку time-to-live датафида в блокчейне.

### 6
Крон-выражение для отправки задается в переменных окружения, чтобы можно было обновить при рестарте сервиса.

Способ: команда devops вносят в config maps. При страте пода проверяем наличие всех параметров.

### 7
Есть флаг для отключения функциональности - при выставлении false значения в блокчейн не отправляются.

Способ: анализ лог-файлов, анализ записей БД на тестовом стенде.
