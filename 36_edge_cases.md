
### 1.
Есть класс Deal (сделка). У него есть множество полей (amount, price, type, template ...).
Сделка формируется в результате вычитки события выпуска токена в блокчейне, по одному из множества смарт-контрактов.

Возникает новое требование: в случае, если это сделка по смарт-контракту с типом RedeemTradeIndex, то нужно получить для него дату redeemSecondDate из базы данных, и передать в сделке. 

Как было сделано:

В класс Deal добавляется новое поле redeemSecondDate.
В методе, где создается объект Deal (который и без того очень раздут), добавляется вызов нового метода fillRedeemSecondDate. 

```java
private void fillRedeemSecondDate(Raw<Deal> r) {
    //...
    Deal deal = ...
    SmartContract smartContract = ...
    if(shouldFillRedeemSecondDate(deal, smartContract)) {
        OffsetDateTime date = getRedeemSecondDate(smartContract);
        deal.setRedeemSecondDate(date);
    }
   }
```

Также добавляется метод shouldFillRedeemSecondDate, где определяется, подходит ли сделка под условия, 
и метод getRedeemSecondDate - где идет запрос в базу данных в таблицу redeem_parameters, где находим нужную дату. 

После изучения материала HW:

Т.к. у нас в целом cделки для разных смарт-контрактов имеют примерно одинаковый набор полей, и такие случаи, как описано выше, редки, можно выделить отдельный класс, например, DealWithOptionals.
Он будет иметь поле "optional" типа map, в формате ключ-значение. 

Сделку без опциональных параметров будем создавать как раньше. А для создания сделки с опцилнальными полями, сделаем новый метод. 


### 2.
Сервис создает и отправляет сообщение о регистрации нового потока данных (датафида) в другой микросервис. Сообщение в формате json, и в коде соответствует классу DatafeedRegistrationMessage.

Сначала для всех типов датафидов поля были одни и те же (название, тип, единицы измерения, период обновления и т.д), а потом начали появляться дополнительные поля для разных типов датафидов.

Например, для типа Qoute стало необходимо добавить признак "Тир". Для этого в  класс DatafeedRegistrationMessage было добавлено поле tear.
Чтобы его заполнить, нужно было сделать запрос в третью систему, получить и распарсить ответ. Поэтому в метод, где заполняются поля DatafeedRegistrationMessage была добавлена ветка:

```
if (datafeed.getType() == QUOTE) {
  // call to 3-d party
}
```

Для датафида типа BidOffer необходимо заполнить тип bid или offer. Для этого нужно распарсить описание датафида по заданным правилам.
Здесь также была добавлена ветка 
```
if (datafeed.getType() == BID_OFFER) {
  // parse datafeed.description
}
```
После изучения материала HW:

Возможно, здесь пока не самый подходящий случай, чтобы добавить отдельный объект наподобие StrangeDatafeedRegistrationMessage, т.к. различных типов немного, и краевые случаи для каждого типа свои собственные. 
Можно сделать отдельные подтипы DatafeedRegistrationMessage для каждого типа датафидов. И каждый заполнять в отдельных методах - так будет более наглядно заполнение полей сообщения каждого типа.


### 3.
Сервис (оракул для блокчейн) принимает сообщения с котировками различных финансовых активов из различных источников.
Котировка - это сообщение в формате json. В нем есть различные поля - название актива, идентификатор системы-источника, но самое главное - это цена актива. Она в рублях, с двумя знаками после запятой. 

Но далее поступают условия, чтобы сервис обрабатывал не только котировки финансовых активов, а в принципе, потоки неких данных для различных инструментов.
Например, поток "событий" со значением "да/нет", поток значений какой-либо процентной ставки (число с двумя знаками после запятой), потом значений коэффициентов (с 4-мя знаками после запятой).

Как было сделано:
Для каждого типа значений датафида создается своя таблица, и свои классы для обработки. 
Получается наглядно, и безопасно, т.к. на уровне базы данных определили тип "значения", и невозможно в таблицу для ставок записать значение коэффициентов.

Но дальше возникли краевые случаи - появились коэффициенты с 10-ю и 2-мя знаками после запятой, и у каждого свой бизнес-смысл.
Тут возникло сомнение, что для них стоит создавать новые таблицы. А вдруг появятся коэффициенты с другой точностью? Или будет коэффициент с такой же точностью, но с другим бизнес-смыслом?

После прочтения материала HW:

Возможно, стоит создать единую таблицу для коэффициентов, где поле "значение" будет иметь 10 (максимально) знаков после запятой.
И сделать тополнительное поле "precision". Так можно будет сделать валидацию в коде, что точность удовлетворяет той, которая требуется.
Здесь нет отдельного объекта, который обрабатывает краевые случаи, но сама структура таблицы их в себе учитывает.
