Найдите в своих (или чужих) проектах и опишите абстракции -- что во что в соответствующих частях кода отображается.


### 1.
Есть сервиc A, который принимает сообщение от системы F (по протоколу FIX), перекладывает в объект своей доменной модели, потом перекладывает в объект доменной модели сервиса B, 
и отправляет в B посредством его API.

В это пункте опишу абстракцию, которая отображает сообщение FIX в объект доменной модели A (нашего сервиса).

Изначально абстракция была такова, что сообщение FIX отображалось в класс QuoteEntity просто перекладыванием полей. Единственным требованием было сохранить нужные нам бизнес-значения.
`8=FIX.4.4|35=W|49=A|56=F|34=5|52=20260324-03:06:27.316| ...` =>

```java
public class QuoteEntity {
    String instrumentId;
    System source;
    LocalDateTime valueDate;
    BigDecimal value;
    //...
```
(объект QuoteEntity сохранялся посредством ORM в таблицу quote_datafeed_value, поэтому также можно сказать, что сообщение отображалось в таблицу quote).

Процесс был такой: когда приходило сообщение, оно том же потоке обрабатывалось и отправлялось в систему B через вызов REST.
Но в один момент в системе B случалась деградация производительности, и REST-запрос вместо нескольких секунд стал занимать десятки секунд, вплоть до нескольких минут.
И тут стало понятно еще одно свойство сообщений FIX, которое должно сохраниться в абстракции нашей доменной модели: единственное актуальное сообщение &ndash; это самое новое.
Пока все было хорошо, это свойство автоматически сохранялось, т.к. мы получали новое сообщение и сразу отправляли его. 
Но когда B стал долго отвечать, оно нарушилось, т.к. пока мы отправляли сообщения, они уже устаревали.

Поэтому реализацию абстракции поменяли: сообщение стало отображаться в 2 таблицы:

`actual_quote_datafeed_value` &ndash; запись (после преобразования) самого свежего сообщения на текущий момент;

`quote_datafeed_value` - запись (после преобразования) всех сообщений;

Теперь при приеме сообщения оно просто сохраняется в таблицу quote и actual_quote. 
А дальше в отдельном потоке по расписанию стал вызываться метод, который извлекает из таблицы actual_quote актуальную запись, и отправляет в систему B.
Несмотря на то, что за время отправки успевает прийти несколько новых сообщений, мы всегда отправляем только самое актуальное. 

Теперь абстракция реализуется не конкретным классом QuoteEntity, как изначально, а совокупностью классов, которые обеспечивают нужное нам свойство.

### 2. 
Рассмотрю еще один пример из того же сервиса, что в п.1.
Только в этот раз абстракция будет касаться не сообщения конкретного типа FIX, а в целом всех сообщений, которые принимает этот сервис.
Каждое сообщение в общем несет собой информацию по состоянию какого-то финансового инструмента (Название инструмента + value). 

Рассмотрим абстракцию, которая отображает это "состояние инструмента" в нашу доменную модель.
Какие свойства мы хотим сохранить? Нам важно сохранить информацию о типе данных по конкретному инструменту. Это может быть:

- рубли, если это финансовая котировка
- процентный пункт, если это процентная ставка
- коэффициент с разными количествами знаков после запятой
- boolean (наступило ли какое-то событие)

Чтобы это обеспечить, в базе для каждого типа сообщений будет своя таблица, где value представлено разными типами данных:
```sql
quote_datafeed_value {
  value numeric(18,2); //для денег нам нужно 2 знака после запятой
}
coefficient_datafeed_value {
  value numeric(12,10); // для коэффициентов 10 знаков после запятой
}

flag_datafeed_value {
 value bool; // да или нет
}

CREATE DOMAIN percentage AS NUMERIC(5, 2);
CHECK (VALUE >= 0 AND VALUE <= 100);
rate_datafeed_value {
   value percentage; // для процентной ставки
}
```

### 3.
Сервис получает в блокчейне адрес, а затем преобразовывает в короткий ключ. Адрес имеет размер 40 байт, ключ - 4 байта.

Рассмотрим абстракцию, которая отображает длинный адрес в короткий ключ. Сохранятся ли нужные нам свойства длинного адреса в абстракции короткого ключа?
Свойство, которое нам надо сохранить &ndash; уникальность короткого ключа для разных адресов. Т.е. мы хэшируем длинный адрес, и требуем, чтобы не было коллизий.
Очевидно, что это свойство когда-то будет нарушено. При кол-ве длинных адресов 77 тыс. в нашей базе данных мы с вероятностью 50% получим дубликат.
Но у нас нет требования поддерживать абсолютно каждый длинный ключ! Поэтому мы можем просто пропустить адрес, который дал ключ, который уже есть в нашей базе.

В итоге наша абстракция отображает длинный адрес в уникальный короткий ключ, либо ни во что.
Технически, мы пробуем сохранить адрес + ключ в БД, и если получаем ошибку при попытке сохранить уникальное поле, то просто пропускаем данный адрес. 

### 4.
В коде делается вызов метода third-party библиотеки, который возвращает объект из этой библиотеки. Мы преобразуем его к нашей объектной модели, оставляя только те поля и операции, которые нам нужны.

Например:
В коде вызываем метод, который возвращает объект SmartContractState, который хранит состояние смарт-контракта.
Нам же интересны только операции погашения по финансовому инструменту, который связан с данным смарт-контрактом. Поэтому в данном отображении сохраняются нужные нам свойства: 

```java
public abstract class RedeemOperation {
   String getRedeemId();
   OperationType getOperationType();
   String getSum();
}

List<RedeemOperation> getRedeemOperations(SmartContractState st) {...}
List<RedeemOperation> getIndexRedeemOperations(SmartContractState st) {...}
List<RedeemOperation> getBondRedeemOperations(SmartContractState st) {...}
List<RedeemOperation> getCouponRedeemOperations(SmartContractState st) {...}
```

### Как вы понимаете определение Дейкстры применительно к своей практике, как сможете его применять.

Если цель абстракции -- быть максимально точной, то на практике буду стремиться к тому, чтобы четко видеть, что во что отображается в моем коде, и какие именно свойства надо сохранить при отображении. 
Не буду стараться делать максимально общее отображение, чтобы захватить все возможные свойства "на всякий случай".

Не добавлять новые классы из Service Layer Abstraction просто потому, что так принято в Spring, когда существующие классы сильно разрослись.
