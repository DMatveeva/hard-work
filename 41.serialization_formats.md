_Присутствуют ли важные структуры данных непосредственно в коде вашего проекта или только в процессах сериализации?_


В основном, все структуры данных присутствуют непосредственно в коде в виде DTO, на Java.
Но есть 2 интеграции, где не так. 

#### 1.
Первая интеграция &ndash; по протоколу EFX. 

Когда приходит сообщение, оно десериализуется в объект quickfix.Message с помощью библиотеки quickfix.
В качестве IDL выступает документ data dictionary в формате xml. По нему также происходит валидация входящего сообщения. 
Но затем объект уже в коде приложения quickfix.Message парсится и преобразуется в структуру QuoteDto, которая уже непосредственно прописана в коде. 

В данном случае в сообщении не содержится id-шек, в нем содержиться название финансового инструмента, к которому относится котировка. 
Далее в коде через ORM ищем продукта с этим названием. Название продукта в данном случае уникально, и само выступает как id. 

#### 2.
Во второй интеграции используем сообщение в формате xml, которое генерируется из xsd через JAXB.

Оно формируется так: есть MessageMapper, который получает на вход объект MеssageEntity (Entity, созданный ORM-ом), а затем преобразует его в объект типа Element (сгенерированного автоматически по xsd). 

В приниципе, пока поля сообщения были простыми, string/boolean/int, все было просто. 
Но потом, например, понадобилось поле, которое может быть и датой-временем, и пустым. 
Тут уже пришлось определить пользовательский тип в xsd, написать в маппере методы, которые преобразовывают поля типа OffsetDateTime из Java в поля типа ```JAXBElement<String>```.

_Как можно строить REST-API, не используя id-шки в качестве идентификаторов в глобальном хранилище данных._

#### 1. 

Может быть, один из вариантов не использовать id-шки в качестве идентификаторов, это передавать в сообщении не просто id, а набор тех полей, которые необходимы для расчетов. 
Допустим, мы принимаем сообщение с ценой какого-то финансового продукта. Нам присылают json, где есть цена и id продукта. 
```
{
  "value": 1.908,
  "product_id": "12345", -- id в базе
  ...
}
```
=>
Если передавать все необходимые для расчета поля, он преобразуется как-то так:
```
{
  "value": 1.908,
  "product": {
    "name": "A1CDM8", -- имя продукта
    "basis": "index"
  }
  ...
}
```

#### 2. 

Читала, что в domain-driven design используются агрегаты. 

Насколько понимаю, в этом случае, когда приходит json c id-шкой, из базы загружается не только один объект, а все связанные с ним объекты, в которых непосредственно прописана бизнес-логика. 
В этом случае в первый раз надо будет загрузить по fk связанные объекты, но затем будем работать с ними без обращения в БД. 
А после всех вычислений, результат можно сохранить в БД используя plain sql, а не ORM.

***
Мне кажется, нашего проекта тоже касается проблема с id-шками.  
У нас примерно так:

Приходит json-сообщение по REST. Jackson библиотека сразу его преобразует в объект DTO. 
Затем, например, мы по id-шкам из сообщения получаем объекты из БД через ORM, проверяем что они существуют или валидируем их поля.

И, например, проверить существование записи с заданным id в БД было бы намного быстрее через plain sql. 
Но встроить этот plain sql в наше приложение на Spring и протестировать занимает больше времени, чем пойти по проторенной дороге и просто в репозитории написать сигнатуру метода (который реализует сам Spring Data), а потом проверить, что возвращаемый объект равен null.
Поэтому многие вещи получаются намного менее производительными, чем можно.
